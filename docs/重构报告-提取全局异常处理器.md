# StudyFlow 重构报告：提取全局异常处理器

## 一、原代码问题分析

### 问题描述

`TaskController` 中 `createTask`、`updateTask`、`deleteTask`、`updateStatus` 四个方法包含**完全相同**的异常处理模板代码：

```java
// 以下代码在 Controller 中出现了 4 次，仅 service 调用不同
try {
    xxxService.someMethod(...);
    return ResponseEntity.ok(...);
} catch (IllegalArgumentException e) {
    Map<String, String> error = new HashMap<>();
    error.put("message", e.getMessage());
    return ResponseEntity.badRequest().body(error);
}
```

### 违反的设计原则

| 原则 | 具体违反 |
|------|---------|
| **DRY（Don't Repeat Yourself）** | 相同的异常处理模板复制 4 次 |
| **单一职责原则（SRP）** | Controller 既要处理 HTTP 路由，又要负责异常格式化和响应码映射 |
| **开闭原则（OCP）** | 新增异常类型需要修改每个 Controller 方法的 catch 分支，而不是通过扩展新类来实现 |

### 可维护性风险

- **改动遗漏**：修改错误响应格式时，需要在 4 处同步修改，遗漏任一处就会导致错误格式不一致
- **扩展成本**：每新增一个 Controller 方法，就要复制一次完整的 try-catch 模板

---

## 二、重构目标

1. **消除重复**：将异常处理逻辑集中到一处，消除 4 处完全相同的 try-catch 代码
2. **职责分离**：Controller 只负责 HTTP 路由和调用 Service，异常处理由独立模块负责
3. **遵循 OCP**：新增异常类型时，只需在 GlobalExceptionHandler 中添加一个方法，无需修改任何 Controller
4. **引入合适的抽象机制**：采用 Spring `@RestControllerAdvice` + `@ExceptionHandler` 实现**模板方法模式（Template Method Pattern）**

---

## 三、重构采用的设计模式

### 模板方法模式（Template Method Pattern）

`@RestControllerAdvice` 结合 `@ExceptionHandler` 本质上是模板方法模式在 Spring 框架中的体现：

```
┌─────────────────────────────────────────────────────┐
│           GlobalExceptionHandler                     │
│                                                     │
│  模板（Template）：                                   │
│  ┌─────────────────────────────────────────────┐    │
│  │ @ExceptionHandler                           │    │
│  │ ResponseEntity<Map<String,String>> 处理流程   │    │
│  │   1. 构建错误 Map                            │    │
│  │   2. ResponseEntity.badRequest().body(map)  │    │
│  └─────────────────────────────────────────────┘    │
│                                                     │
│  变化点（Variation Points）：                         │
│  ┌─────────────────────────────────────────────┐    │
│  │ handleIllegalArgument(IllegalArgumentException)│ │
│  │ handleXxxException(XxxException)  ← 未来扩展  │    │
│  └─────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────┘
```

**模式角色对应关系**：

| 模板方法角色 | 对应实现 |
|-------------|---------|
| 抽象模板 | `@RestControllerAdvice` 类 |
| 模板方法 | Spring 框架根据异常类型自动匹配 `@ExceptionHandler` |
| 具体变化点 | `handleIllegalArgument()` 等各异常处理方法 |
| 扩展方式 | 新增 `@ExceptionHandler` 方法，对修改封闭，对扩展开放 |

### 4.3 重构前后代码对比（以 `createTask` 为例）

**重构前**（Controller 承担路由 + 异常处理双重职责）：

```java
@PostMapping
public ResponseEntity<?> createTask(@RequestBody Task task) {
    try {
        Task created = taskService.createTask(task);
        return ResponseEntity.ok(created);
    } catch (IllegalArgumentException e) {
        Map<String, String> error = new HashMap<>();
        error.put("message", e.getMessage());
        return ResponseEntity.badRequest().body(error);
    }
}
```

**重构后**（Controller 只负责路由，异常处理统一由 GlobalExceptionHandler 接管）：

```java
@PostMapping
public Task createTask(@RequestBody Task task) {
    return taskService.createTask(task);
}
```

**新增 GlobalExceptionHandler**（异常处理的唯一归属点）：

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(IllegalArgumentException.class)
    public ResponseEntity<Map<String, String>> handleIllegalArgument(IllegalArgumentException e) {
        Map<String, String> error = new HashMap<>();
        error.put("message", e.getMessage());
        return ResponseEntity.badRequest().body(error);
    }
}
```

## 六、扩展性对比

### 场景：新增一种异常类型 `TaskNotFoundException`

**重构前的改动**（违反 OCP，需修改已有代码）：

```java
// 需要在 4 个 Controller 方法的 catch 块中都增加：
} catch (TaskNotFoundException e) {
    return ResponseEntity.status(HttpStatus.NOT_FOUND)
            .body(Map.of("message", e.getMessage()));
}
// 每个方法都要改，共改 4 处
```

**重构后的改动**（遵循 OCP，只需扩展新类/新方法）：

```java
// 只需在 GlobalExceptionHandler 中新增一个方法，Controller 零改动：
@ExceptionHandler(TaskNotFoundException.class)
public ResponseEntity<Map<String, String>> handleNotFound(TaskNotFoundException e) {
    return ResponseEntity.status(HttpStatus.NOT_FOUND)
            .body(Map.of("message", e.getMessage()));
}
```

## 九、重构决策依据

| 评估维度 | 选择 GlobalExceptionHandler 的理由 |
|---------|----------------------------------|
| **当前痛点** | 4 处完全相同的 try-catch 是肉眼可见的重复，是当前代码库最大的 DRY 违反 |
| **改动成本** | 仅涉及 2 个文件（新增 1 个 + 修改 1 个），不触碰 Service/Mapper 层 |
| **风险** | 行为完全不变，现有 61 个测试全部通过，无回归风险 |
| **引入的抽象** | `@RestControllerAdvice` 是 Spring 框架原生支持的机制，本质为模板方法模式，无需额外学习成本 |
| **扩展收益** | 未来新增异常类型只需加一个方法，Controller 零改动，符合开闭原则 |
| **项目规模匹配** | 适合当前单实体 CRUD 项目规模，不过度设计 |
