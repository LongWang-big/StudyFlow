# AI 辅助重构个人反思

## 一、使用的 Prompt 与 AI 输出摘要

### 1. 代码评审阶段

**Prompt：** "请对以下代码进行软件构造层面的评审，重点指出：可扩展性问题、可测试性问题、违反单一职责或开闭原则的部分、可以采用的重构方向"

**AI 输出：** 产出了结构化的评审报告，识别出 9 个具体问题，包括 4 处重复 try-catch、String 类型不安全、贫血模型、`new Date()` 时间耦合等，最终归纳为 8 个重构方向的优先级表格。

### 2. 方案比较阶段

**Prompt：** "说明最终选择理由，包括：对现有功能影响范围、是否需要新增抽象层、测试是否容易迁移、是否符合当前项目规模"

**AI 输出：** 在方案 A（提取全局异常处理）和方案 B（枚举化 + 状态机）之间做了四个维度的对比分析，最终选择方案 A，理由是"当前痛点是重复代码，不是结构不足"。

### 3. 补齐测试阶段

**Prompt：** "重构前先补齐缺失测试"

**AI 输出：** 识别出 Controller 层零测试的缺口，新建 `TaskControllerTest.java`，用 `@WebMvcTest` + MockMvc 编写了 19 个测试，覆盖全部 7 个接口的正常路径、异常路径和边界条件。

### 4. 重构执行阶段

**Prompt：** "按照方案 A 执行重构，在测试保护下完成。每次提交只做一种类型的修改，保证对外行为不变，每完成一步都运行测试"

**AI 输出：** 将重构拆分为 3 个原子步骤，每步只做一种修改并运行测试验证，最终 Controller 从 108 行简化为 82 行，异常处理逻辑集中到 GlobalExceptionHandler（19 行）。

## 二、遇到的问题与解决过程

### 问题 1：AI 跳过评审直接动手重构

当我发出代码评审请求时，AI 直接开始写重构代码（创建 GlobalExceptionHandler 文件）。我立即中断并纠正："我是让你选择一个，不是让你重构。"

**解决：** 重新明确指令，AI 随后产出了两套方案的对比分析，而不是直接实现。这说明使用 AI 时需要**明确区分"分析"与"执行"的边界**，否则 AI 会默认跳到实现。

### 问题 2：测试编译失败——`@MockBean` 包名错误

编写 Controller 测试时，`import org.springframework.boot.test.mock.bean.MockBean` 编译报错，提示包不存在。

**排查过程：**
1. 检查 `pom.xml`：`spring-boot-starter-test` 依赖已声明 ✅
2. 检查本地 Maven 仓库：`spring-boot-test-2.7.18.jar` 存在 ✅
3. 检查 jar 内容：`jar tf` 发现 `MockBean.class` 实际位于 `org.springframework.boot.test.mock.mockito` 包下

**解决：** 将 import 从 `...mock.bean.MockBean` 改为 `...mock.mockito.MockBean`，编译通过。

**教训：** AI 生成的代码依赖特定 Spring Boot 版本的包结构，实际项目中的版本可能不同，需要**以编译结果为准，而非信任 AI 的 import 路径**。

### 问题 3：未找到构建工具

运行测试时 `./mvnw test` 报 "no such file or directory"，项目没有 Maven Wrapper。

**解决：** 检查项目根目录发现有 `pom.xml` 但无 `mvnw`，改用系统安装的 `mvn test` 命令。这属于项目环境问题，与 AI 生成的代码无关，但说明 AI 默认使用 `./mvnw` 是基于常见约定，实际需要根据项目调整。

## 个人反思

### 一、这次重构让我真正理解了"重复代码"的代价

在写这段代码的时候，我其实知道每个 Controller 方法里都有 try-catch，但当时觉得"能跑就行"。直到这次做代码评审，把四个方法并排摆在一起看，才真正意识到：这不是"写了四遍"，而是**埋了四颗地雷**——将来改错误格式的时候，漏改一处就是 bug。

这次经历让我养成了一个习惯：写完一段逻辑后，如果发现自己在另一个地方写了几乎一样的代码，就停下来想一想，是不是应该提取出来。重复代码最可怕的不是多写了几行，而是**它会让你觉得"改一个地方就够了"，但实际上要改四个地方**。

### 二、测试不是重构的"附属品"，而是"前提条件"

以前我对测试的理解是"写完代码顺手补一个"。但这次重构之前，我先花了时间补了 19 个 Controller 测试。当时觉得有点浪费时间——又没有改代码，为什么要先写测试？

后来真正开始重构的时候，每改完一步就跑一次测试，看到 61 个绿勾，心里很踏实。如果没有这些测试，我大概会在移除 try-catch 的时候反复纠结："这样做会不会改变响应格式？""Spring 会不会把异常直接抛给客户端？"但有了测试之后，跑一遍就知道答案，不用猜。

**我现在觉得，测试不是重构完成后的验收工具，而是重构开始前的安全网。** 没有安全网的重构，就是闭着眼睛走钢丝。

### 三、AI 能帮我写代码，但不能帮我做判断

这次和 AI 协作下来，我最大的感受是：**AI 很擅长"怎么做"，但不擅长"该不该做"**。

有几个时刻我印象很深：

- 我让 AI 做代码评审，它直接开始写重构代码——它默认"发现问题→修复问题"，但没有意识到我需要先分析、再选择、再执行
- 我让它选择一个方案，它给了很好的对比分析；但如果没有我明确要求"至少两个候选"，它可能就直接推荐一个然后动手了
- 测试里 `@MockBean` 的 import 路径写错了，AI 不会意识到这个问题——因为它是基于训练数据生成的，不知道我们项目用的 Spring Boot 版本

这些经历让我意识到：**和 AI 协作，人的核心价值不是写代码，而是"在正确的时机提出正确的问题"**。我需要告诉 AI 我要什么、不要什么、什么时候停下来。如果我只是说"帮我重构"，AI 会很快给我一个结果——但那个结果不一定是我真正需要的。

### 四、小步提交比大步提交更安全

这次重构我分了三个 commit：补测试、新增处理器、移除旧代码。每一步都很小，每一步都可以独立回滚。

如果我图省事，把三个步骤合成一个 commit，虽然最终结果一样，但**我失去了"回到中间状态"的能力**。比如万一移除 try-catch 后出了问题，我只能回退到"没有测试、没有处理器"的最初状态，而不是"有测试、有处理器、还没移除旧代码"的中间状态。

这让我联想到 Git 的设计哲学：**commit 不是"我做了什么"的记录，而是"我可以回到哪里"的路标**。路标越密集，回退的成本越低。

### 五、总结

这次重构虽然改动量不大（净减少 7 行代码），但我学到了比代码更重要的东西：

1. **重构不是改代码，是改结构**——代码变少了，但职责变清晰了
2. **测试是重构的信心来源**——没有测试的重构，就是在赌
3. **AI 是工具，不是决策者**——用好 AI 的关键是知道什么时候让它停下来
4. **小步快跑比大步跨越更可控**——每个 commit 只做一件事
