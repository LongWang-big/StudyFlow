# 重构设计方案比较说明

## 一、待解决的原始问题

经过对 `TaskController` 和 `TaskServiceImpl` 的代码评审，识别出以下核心设计问题：

| 编号 | 问题 | 严重程度 |
|------|------|---------|
| P1 | Controller 中 4 处完全相同的 try-catch 异常处理模板 | 高（DRY 违反） |
| P2 | `status` 和 `priority` 使用 String 而非枚举，编译器无法捕获非法值 | 中（类型安全） |
| P3 | 状态流转规则硬编码在 if 语句中，只有一条规则但分散在 Service 里 | 中（可扩展性） |
| P4 | `new Date()` 直接调用，测试无法精确控制时间边界 | 低（可测试性） |

本报告针对 **P1（重复异常处理）** 和 **P2+P3（类型安全 + 状态机）** 两个方向提出两套候选方案。

---

## 二、方案 A：提取全局异常处理器

### 2.1 设计思路

将 Controller 中重复的 try-catch 模板抽取为 Spring `@RestControllerAdvice`，利用框架的异常拦截机制统一处理 `IllegalArgumentException`。

### 2.2 涉及改动

```
新增：handler/GlobalExceptionHandler.java（19 行）
修改：controller/TaskController.java（移除 4 处 try-catch，-35 行，+9 行）
不变：TaskServiceImpl / TaskMapper / Task / 测试
```

### 2.3 重构后结构

```
TaskController
├── createTask()        → 直接返回 taskService.createTask(task)
├── updateTask()        → 直接返回 taskService.updateTask(id, task)
├── deleteTask()        → 直接调用 taskService.deleteTask(id)
├── updateStatus()      → 直接调用 taskService.updateTaskStatus(id, status)
└── getTaskById()       → 保留 null 判断返回 404（非 IllegalArgumentException）
    getTasks()          → 保留筛选逻辑（不涉及异常）
    getStats()          → 保持不变

GlobalExceptionHandler              ← 新增
├── handleIllegalArgument()         → 拦截 IllegalArgumentException → 400
└── 未来扩展：handleXxxException()  → 新增异常类型时加一个方法即可
```

### 2.4 优缺点

| 维度 | 评价 |
|------|------|
| 改动范围 | 极小：新增 1 文件，修改 1 文件，不触碰 Service/Entity |
| 风险 | 接近零：行为完全不变，61 个测试全部通过 |
| 类型安全 | ❌ 不解决：status/priority 仍是 String |
| 状态机 | ❌ 不解决：流转规则仍是 if 硬编码 |
| 扩展性 | ✅ 新增异常类型只需加一个 @ExceptionHandler 方法 |
| 项目规模匹配 | ✅ 适合当前单实体 CRUD 规模 |

---

## 三、方案 B：枚举化 + 状态机提取

### 3.1 设计思路

将 `status` 和 `priority` 从 String 改为 Java 枚举，同时在枚举中内聚合法的状态转移规则，消除 Service 层硬编码的 if 判断。

### 3.2 涉及改动

```
新增：entity/TaskStatus.java        → 枚举，含 canTransitionTo() 方法
新增：entity/TaskPriority.java      → 枚举，含 requiredDeadline 标记
修改：entity/Task.java              → status/priority 字段类型 String → 枚举
修改：service/TaskServiceImpl.java  → 字符串比较改为枚举比较，状态校验委托给枚举
修改：service/TaskServiceImplTest.java → mock 和断言中的字符串改为枚举值
新增：MyBatis 配置                  → 枚举类型处理器（TypeHandler）
```

### 3.3 重构后结构

```java
public enum TaskStatus {
    TODO,
    IN_PROGRESS,
    COMPLETED;

    private static final Map<TaskStatus, Set<TaskStatus>> TRANSITIONS = Map.of(
        TODO,          Set.of(IN_PROGRESS),
        IN_PROGRESS,   Set.of(COMPLETED, TODO),
        COMPLETED,     Set.of()              // 不可回退
    );

    public boolean canTransitionTo(TaskStatus target) {
        return TRANSITIONS.getOrDefault(this, Set.of()).contains(target);
    }
}

public enum TaskPriority {
    HIGH(true),       // 高优先级必须设置截止时间
    MEDIUM(false),
    LOW(false);

    private final boolean requiresDeadline;
    // ...
}
```

### 3.4 优缺点

| 维度 | 评价 |
|------|------|
| 改动范围 | 较大：涉及 Entity + Service + Test + MyBatis 配置 |
| 风险 | 低，但需处理 MyBatis 枚举映射 |
| 类型安全 | ✅ 编译器捕获非法值（`setPriority("HIGHT")` 直接报错） |
| 状态机 | ✅ 转移规则集中在枚举 TRANSITIONS 表中，清晰可查 |
| 扩展性 | ✅ 新增状态只需加枚举值 + 更新转移表 |
| 项目规模匹配 | ⚠️ 当前只有 3 个状态、1 条规则，有过度设计风险 |

## 四、方案对比总表

| 对比维度 | 方案 A：全局异常处理 | 方案 B：枚举化 + 状态机 |
|---------|--------------------|-----------------------|
| **解决的问题** | P1（重复异常处理） | P2（类型安全）+ P3（状态机） |
| **未解决的问题** | P2、P3、P4 | P1、P4 |
| **改动文件数** | 2 | 5-6 |
| **新增类数** | 1 | 2 枚举 + 可能的 TypeHandler |
| **测试迁移量** | 零（现有测试不变） | 需修改测试中所有 String 为枚举值 |
| **引入的设计模式** | 模板方法（@RestControllerAdvice） | 状态模式（枚举内聚转移表） |
| **回归风险** | 接近零 | 低（需注意 MyBatis 映射） |
| **消除的重复** | 4 处 try-catch（-35 行） | 散落各处的字符串比较 |
| **可扩展性提升** | 新增异常类型加 1 个方法 | 新增状态加 1 个枚举值 + 更新表 |
| **长期收益** | 中（解决当前重复） | 高（奠定类型安全基础） |
| **短期投入** | 低（30 分钟可完成） | 中（需 1-2 小时，含配置调试） |

## 五、最终选择：方案 A

### 5.1 选择理由

#### （1）对现有功能影响最小

方案 A 只涉及异常处理层，不触碰业务逻辑。重构前后的 HTTP 响应（状态码、响应体格式、错误消息）完全一致，61 个测试全部通过。方案 B 需要修改 Entity 字段类型，会牵动 Service 层和测试中的所有相关代码，影响面更大。

#### （2）不需要新增抽象层

方案 A 利用 Spring 框架原生的 `@RestControllerAdvice` 机制，不需要自造接口、基类或配置。方案 B 需要引入两个枚举类，还需要配置 MyBatis 的枚举类型处理器，增加了项目的概念数量。

#### （3）测试完全不需要迁移

方案 A 的现有 42 个 Service 测试和 19 个 Controller 测试原封不动全部通过。方案 B 需要将测试中所有 `setStatus("TODO")` 改为 `setStatus(TaskStatus.TODO)`，涉及约 30 处修改。

#### （4）与当前项目规模匹配

当前项目是单实体 CRUD 应用，状态流转规则只有一条（COMPLETED → TODO）。在这个规模下，引入完整的枚举状态机属于过度设计。方案 A 精准解决当前最显眼的重复问题，不过度引入超出项目需要的复杂度。

#### （5）为后续迭代保留空间

方案 A 不排斥方案 B。未来项目规模增长、规则增多时，可以在方案 A 的基础上再实施枚举化，两者互不冲突。

### 5.2 选择矩阵

| 决策因素 | 权重 | 方案 A | 方案 B |
|---------|------|--------|--------|
| 改动成本 | 高 | ✅ 2 个文件 | ⚠️ 5-6 个文件 |
| 回归风险 | 高 | ✅ 接近零 | ⚠️ 需处理映射配置 |
| 解决当前痛点 | 高 | ✅ 消除 4 处重复 | ❌ 不涉及异常处理 |
| 长期扩展性 | 中 | ⚠️ 中等 | ✅ 高 |
| 测试迁移量 | 中 | ✅ 零迁移 | ⚠️ ~30 处修改 |
| 项目规模匹配 | 中 | ✅ 匹配 | ⚠️ 略超前 |

---

## 六、结论

方案 A 和方案 B 解决的是**不同层面**的问题，而非同一个问题的两个解法：

- **方案 A** 解决的是**代码重复**（DRY 违反）—— 这是当前最直接、最高频的维护痛点
- **方案 B** 解决的是**类型安全和状态管理**—— 这是更深层的结构问题，但当前规模下尚未成为实际痛点

选择方案 A 的核心逻辑是：**先止血，再治本**。重复代码每天都在增加维护成本（每加一个 Controller 方法就多一次复制粘贴），而类型安全问题在当前 3 个枚举值的规模下尚未造成实际错误。因此优先处理投入产出比更高的方案 A，是符合当前项目阶段的务实选择。
